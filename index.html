The problem

When an agent moves money, the bank can't tell if it's trustworthy

Every autonomous AI agent payment lands in the manual review queue. Agent autonomy dies at the compliance wall — and there is no portable, verifiable way to vouch for a machine. The root problem: trust is stored in a vendor's database, checked by an API, and enforced in application code above the money. That is skippable.

Typical agent-trust stack

Enforced a layer up

Trust is a row in a vendor's database
Checked by an API call the relying party must trust
Enforced in application code, above the money
Revocation = a DB flag every integrator must honor
MeshCredit

Enforced at consensus

Trust is a 6-dimension XLS-70 certificate on the ledger
Read directly from the ledger, no API key, no middleman
Enforced by validators — the payment cannot settle without it
CredentialDelete → denied everywhere in one ~4 s close

Treasury is a single signer today; multi-sig / HSM in production.

The product

The multi-dimensional certificate

The verdict is minted as an XLS-70 certificate carrying a deterministic Bronze→Platinum tier and a first-class Confidence Level. Six dimensions, each with a named issuer and real evidence — independently checkable by anyone. The product is the certificate, not any single payment.

Principal model

Pluggable root of accountability

The subject is the AI agent. The principal is pluggable: Organization → KYB (Swiss Zefix registry); Individual → KYC; Parent-agent → delegation (capped by delegator's budget); Pseudonymous → capped at lowest tier. Anti-impersonation: a principal claim is valid only if the named principal counter-signs the agent key.

Honesty

Zefix: cached real public-registry data — the live API requires credentials. World ID: simulator / staging — an orb needs a physical device. Treasury: single signer today; multi-sig / HSM in production. Thick-file agent: uses real seeded on-chain history.

What consensus enforces: credential existence + gating via DepositAuth / DepositPreauth + XLS-80 Permissioned Domains. Consensus does not validate the score; MeshCredit enforces dimension integrity via independently-checkable evidence.

Challenge fit — all four Pillar-3 angles

The judging rubric weights Viability at 40%. Every listed primitive is live on Mainnet, and the enforcement path (DepositAuth/DepositPreauth + XLS-70 + XLS-80) is fully operational. MeshCredit directly addresses all four Pillar-3 example angles.

Angle 1

KYA identity

Six-dimension certificate issued as an XLS-70 Credential. D1 anchors to did:xrpl; D3 code-provenance hash is recomputable; D6 OFAC gate is a hard dependency. Credential-scoped venues via XLS-80 Permissioned Domains.

Angle 2

Agent wallets with spending policies

Effective ceiling = min(tier ceiling, principal remaining delegation budget). A Bronze agent is hard-capped at $100. A parent-agent's delegation cap propagates down. The limit is enforced at consensus, not in application code.

Angle 3

Regulated settlement incl. x402

RLUSD escrow (XLS-85 TokenEscrow) provides the regulated settlement leg. A lightweight x402 compliance gate screens payments at the request layer before they touch the ledger. AML/OFAC runs as a hard gate in D6.

Angle 4

Agent-to-agent delegation

Parent-agent principal type in the pluggable model: a parent agent counter-signs the child's key, passing a delegation budget. The child is capped at the parent's remaining budget. Recursively checkable on-chain.

40% Viability weight — most primitives are Mainnet-ready. Novelty weight 15%: enforcement-at-consensus is the distinction rivals (t54, Experian, ERC-8004) can't match — they enforce a layer up.

Business model

The data flywheel

More certified agents and verified payments make the bureau smarter and stickier. On-chain auditability is the differentiator a private database can't replicate. Industry analogs: credit bureaus (FICO/Experian), fraud consortiums (Early Warning/Sardine), on-chain analytics (Chainalysis).

Revenue model

Two-sided network

Relying parties (banks, PSPs, venues) pay per verification. Agents and operators onboard to unlock gated rails. The credential is the entry ticket — issued once, read everywhere.

The flywheel

Data → calibration → moat

More on-chain behavioral data → empirically calibrated tier thresholds → graph-scale fraud detection → negative-event sharing (consortium) → reputation portability. A Bronze agent that settles cleanly becomes Gold — auditable because XRPL is public.

Path to Mainnet

Most primitives are live

XLS-70, DepositAuth/DepositPreauth, XLS-80, XLS-85, DEX, did:xrpl, RLUSD — all live on Mainnet. One flag (asfAllowTrustLineLocking) separates the RLUSD escrow leg. Demonstrates 40% Viability criterion directly.

Go-to-market: start with AI procurement agents in regulated corridors (CH → NG); expand to any agentic workflow requiring a portable, on-chain trust anchor.
